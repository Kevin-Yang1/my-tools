# Agent Notes

这个文件是给 AI 看的。你在这个仓库里工作的目标只有一个：把个人常用脚本和小工具整理好，保持简单、可维护、可复用。

## 先做什么

- 先读 `README.md` 和相关目录，再决定改哪里。
- 只改和当前任务有关的文件，不要顺手重构整个仓库。
- 如果信息不够，先停下来说明缺什么，不要猜。

## 仓库分工

- `scripts/` 放可执行脚本，适合轻量、临时、可重复使用的工具。
- `tools/` 放相对完整的小工具，每个子目录对应一个独立功能。
- `docs/` 放说明、例子和设计记录。

## 安装约定

- `install.sh` 会扫描 `scripts/` 下所有可执行文件并安装到 `~/.local/bin`。
- 命令名规则是：去掉常见脚本扩展名，再把 `_` 换成 `-`。
- 如果某个脚本需要短别名，直接在 `install.sh` 里补映射。
- 新增脚本后，要保证它是可执行的，否则不会被安装。

## 写法要求

- 优先最小实现，不要过度抽象。
- 文件名要直接表达用途，避免 `test.py`、`tmp.sh` 这类模糊命名。
- 默认把脚本写成可复用版本，参数化输入，不要把路径和常量写死。
- 新增工具时，最好在工具目录补一个简短 README 或使用说明。
- 默认使用 ASCII；只有在已有内容或实际需要下再引入非 ASCII。

## 每个工具都要交代清楚

- 它做什么。
- 输入和输出是什么。
- 依赖哪些环境或第三方包。
- 怎么运行。
- 会不会改文件、删文件、联网。
- 失败时最常见的原因是什么。

## 依赖与实现

- Python 优先用独立虚拟环境或明确依赖文件。
- Node 优先提供 `package.json`。
- Shell 脚本尽量兼容常见 `bash`。
- 不要为了一个脚本引入复杂工程结构。

## 安全边界

- 默认不要做破坏性操作，包括批量删除、覆盖原始数据和不可逆修改。
- 涉及磁盘、网络、Git 提交、外部 API 的动作，先确认范围再动手。
- 写文件前先检查目标是否已存在，避免误覆盖。

## 提交标准

- 提交要小而明确，信息直接说明改了什么。
- 一次只做一类变更，方便回滚和审查。
- 推到 GitHub 前，先确认 README 和忽略规则是否完整。

## 何时拆仓库

- 这个工具开始变大了。
- 其他人需要长期独立使用。
- 需要单独的 Release、Issue 或 CI。

满足这些条件时，应该建议拆成独立仓库，而不是继续堆在这里。
